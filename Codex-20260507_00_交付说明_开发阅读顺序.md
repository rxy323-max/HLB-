# Codex-20260507 交付说明与开发阅读顺序

> 本文件用于说明 Codex 输出与原始需求文档之间的关系。  
> 结论：Codex 输出不是替代原始字段明细，而是作为 **Non-Individual 进件流程的整合方案、交互重构建议和开发阅读导航**。

## 0. 本轮结论

开发不应该只看 Codex 精炼稿，也不需要在“原始版本”和“Codex 版本”之间二选一。正确交付方式是给开发一个阅读包：

- **Codex-20260507 系列**：用于理解产品流程、页面顺序、现网继承方式、进件体验重构。
- **原 V2/Excel/原始 FSD**：用于查询字段定义、枚举、必填、接口来源、校验、材料、后续风险/放款规则。

也就是说，Codex 输出是“开发入口和交互整合说明”，不是“字段明细替代稿”。上一轮带 `V3` 的文件可视为 Codex 中间稿；后续正式识别统一使用 `Codex-YYYYMMDD_序号_主题`，不再用 `v1/v2/v3` 表示 Codex 产物。

## 2. 是否需要给开发两个版本？

建议给开发一个“阅读包”，而不是只给某一个文件：

1. **先读 Codex 交付说明**
   明确 Non-Individual 不是单独新系统，而是并入现有 HP 申请功能。

2. **再读 Codex 进件整合方案**
   理解推荐流程、导航继承、页面职责和交互调整。

3. **再回到 V2 明细文档和 Excel 字段表**
   查字段、枚举、必填、接口来源、校验、历史规则细节。

4. **遇到冲突时按以下原则处理**
   - 字段定义、枚举、校验格式：以 V2 明细/Excel/原始 FSD 为准。
   - 页面顺序、用户体验、进件步骤：以 Codex-20260507 方案为准。
   - 风险、CED、放款后续工作台：以各自原始 FSD 为准，Codex 只描述进件侧依赖。

## 4. Codex 文件和 V2 文件的关系

Codex 文件主要做三件事：

1. **重排流程**
   原 V2 中“阶段零：申请入口与渠道识别”调整为“先锁定企业主体，再确认申请上下文”。

2. **对齐现网功能**
   Non-Individual 不是独立向导，而是复用现有个人申请的 Navigation、页面容器、列表入口、状态完成标识。

3. **建立字段索引**
   不把所有字段重新抄一遍，而是说明每个页面应该去 V2/Excel/原始 FSD 的哪个章节查详细字段和规则。

## 6. 给开发的一句话说明

Non-Individual 进件并不是新增一套孤立流程，而是在现有 HP 个人申请功能中扩展企业主体、企业画像、股权/UBO、担保结构、企业收入和材料规则；页面容器、导航层级、列表入口、完成状态、风险相关区域应继承现有个人申请体验。
