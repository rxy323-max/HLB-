# Non-Individual 进件系统 - 项目概要

> **文档目的**：快速理解项目背景、核心概念、文件结构和关键决策，配合详细文档一起使用。

## 一、项目背景

**系统名称**：CrediOS - Non-Individual 进件系统  
**所属银行**：马来西亚丰隆银行（Hong Leong Bank, HLB）  
**目标用户**：银行销售人员  
**业务场景**：为企业客户（非个人客户）办理车贷进件

**系统目标**：
- 数字化进件流程，替代纸质申请
- 自动化风险检查（CCRIS/CTOS/AML）
- 整合 Experian API 进行 UBO 穿透
- 符合 BNM（马来西亚国家银行）合规要求

## 二、核心概念速查

### 2.1 实体类型（12类）

企业客户分为 12 种实体类型，系统根据类型展示不同录入字段和流程。

**活跃类型（8种）**：
| 代码 | 英文名称 | 中文名称 | 示例 |
|------|----------|----------|------|
| C | Bumiputera Enterprise | 土著企业 | 马来人独资 |
| D | Non-Bumi Enterprise | 非土著企业 | 华人独资 |
| E | Partnership | 合伙 | 2-20人合伙 |
| F | Limited Liability Partnership | 有限责任合伙 | 现代合伙制 |
| G | Private Limited | 私人有限公司 | Sdn Bhd |
| H | Public Limited | 公众有限公司 | Bhd |
| L | Branch of Foreign Co | 外国公司分支 | 跨国公司 |

**非活跃类型（4种）**：政府、法定机构、金融机构等，流程简化。

### 2.2 角色体系（三维框架）

这是本系统**最易混淆**的部分。需要理解三个独立维度：

| 维度 | 说明 | 关键角色 |
|------|------|----------|
| **维度一：法律身份** | 谁承担债务责任 | 主申请人、Guarantor（担保人）、Non-Guarantor |
| **维度二：所有权身份** | 谁拥有企业、谁控制企业 | Owner、Partner、Shareholder、**UBO**（最终实益控制人） |
| **维度三：签约权力** | 谁有权签署合同 | Signatory（董事/合伙人） |

**关键区别**：
- **UBO ≠ Shareholder**：UBO 是最终控制人（>25%股权或实际控制），Shareholder 是登记股东
- **UBO ≠ Guarantor**：UBO 报送 BNM，Guarantor 承担债务连带责任
- **Signatory ≠ Guarantor**：Signatory 是 BR 授权的签署人，Guarantor 是担保人

**典型场景**：
1. **Sdn Bhd（私人有限公司）**：
   - 主申请人：公司
   - UBO：持股>25% 的董事（自然人）
   - Guarantor：可选（如果添加，自动转手动签署）
   - Signatory：BR 授权的董事

2. **独资企业（C/D类）**：
   - 主申请人：企业（Ownership Type=Own Name）
   - UBO：业主本人（自动识别）
   - Guarantor：不需要（个人承担无限责任）

3. **合伙（E类）**：
   - 主申请人：合伙企业
   - UBO：所有 Partner（强制）
   - Guarantor：所有 Partner（E类强制要求）

### 2.3 UBO 穿透

**UBO（Ultimate Beneficial Owner）**：最终实益控制人，必须 **自然人**。

**穿透逻辑**：
1. **自动抽取**：Experian API 抽取董事/股东（>25%）
2. **树形图展示**：展示企业股权结构
3. **手动 drill-down**：法人股东需要手动穿透（API 限制）

**4种识别模式**：
| 模式 | 适用场景 | 交互方式 |
|------|----------|----------|
| 自动绑定 | 企业+企业穿透到个人 | 系统自动识别 |
| 职位确认 | BR 显示 Owners/Partners | 用户确认是否 UBO |
| 强穿透 | SB 股东>25% | 强制添加为 UBO |
| 豁免 | 保险/外资等 | 无需 UBO |

### 2.4 收入计算器

**用途**：计算申请人偿债能力（DSR - Debt Service Ratio）

**收入聚合规则**：
- ✅ **主申请人 + Guarantor** 收入自动聚合计算 DSR
- ❌ **其他角色**（UBO、Owner、Signatory）收入不计入

**11种收入类型**：
| 收入类型 | 适用人群 | 强制上传文档 |
|----------|----------|--------------|
| Salary/Wage | 受雇人士 | EA Form、工资单 |
| Fixed Allowance | 固定津贴 | 工资单 |
| Commission | 佣金 | 工资单、银行流水 |
| Bonus/OTA | 奖金 | EA Form |
| Pension | 养老金 | Pension Statement |
| Income from Net Profit | 企业主 | Form B/BE、审计报告 |
| Rental Income | 租金收入 | 租赁合同、银行流水 |
| ... | ... | ... |

## 三、系统流程概览

### 3.1 完整流程（12阶段）

```
申请入口 → 渠道识别 → 身份识别 → 实体类型选择 → 企业信息录入 
  → UBO穿透 → 车辆信息选择 → 产品方案配置 → 文档上传 
  → 风险引擎 → CED审批 → BNM报送 → 拨款流程
```

### 3.2 关键决策点

| 决策点 | 影响 |
|--------|------|
| 是否添加 Guarantor？ | 决定签署流程（电子 vs 手动） |
| 实体类型是否正确？ | 决定后续录入字段 |
| UBO 是否识别完整？ | 决定 BNM 报送合规性 |
| 收入计算是否通过？ | 决定是否进入风险引擎 |

## 九、常见问题（FAQ）

### Q1: UBO 和 Guarantor 有什么区别？

**A**: 
- **UBO** 是监管要求（BNM报送），必须是自然人，不一定承担债务
- **Guarantor** 是银行要求（风险控制），承担债务连带责任，可以是个人或企业
- 两者可能重叠（同一个人既是 UBO 又是 Guarantor），但系统必须分开处理

### Q2: 为什么含 Guarantor 的案件要转手动签署？

**A**: HP Product FSD 明确规定：含 Guarantor 案件自动切换为手动签署流程。原因可能是：
1. Guarantor 需要线下签署担保协议
2. 电子签约不适用于多签署方场景（技术限制）
3. 合规要求（担保人必须面签）

### Q3: 收入计算器为什么只聚合主申请人和 Guarantor？

**A**: 因为只有这两类承担债务责任。UBO/Owner/Signatory 不承担债务，所以其收入不计入 DSR 计算。

### Q4: UBO 穿透为什么需要手动 drill-down？

**A**: Experian API 限制：只能抽取董事/股东（>25%），但无法自动穿透法人股东。如果股东是企业（非自然人），需要用户手动查询该企业的股东，直到穿透到自然人。
