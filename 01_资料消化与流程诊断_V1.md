# HLB Non-Individual HP 进件资料消化与流程诊断

> 版本：v1.0  
> 日期：2026-05-07  
> 范围：基于原始 HP Product Sales、Risk、CED、AIOCR、Disbursement FSD，以及 assets 中企业类型、角色、字段、报送规则资料，对现有 V2 文档做产品诊断和流程重构建议。

## 1. 结论摘要

现有 V2 文档已经覆盖了 Non-Individual 进件的大部分业务对象，但流程顺序仍偏“字段罗列型”。它把渠道、分行、销售、企业类型等信息放在入口阶段一次性填写，容易让销售在还没有确认企业主体前就承担过多判断成本。

推荐将进件主线调整为：

```
锁定企业主体 -> 建立申请上下文 -> 补齐企业画像 -> 识别控制人与担保结构 -> 录入收入与风险所需信息 -> 选择车辆/产品/Dealer -> 上传材料并完成 OCR 校验 -> 提交风险/CED
```

其中“银行渠道、分行、销售、Officer、Branch Coverage”应主要来自登录用户、机构主数据、Dealer/Lead 来源和案件分派规则。前端只在发生例外、跨分行、Dealer/Direct 场景或手工改派时让销售确认。

## 2. 资料来源消化

| 来源 | 关键信息 | 对进件流程的影响 |
|---|---|---|
| HP Product Sales FSD | 非个人主申请人识别、ID Type/Enterprise Type 映射、Other Applicants、UBO、收入、HP Line、渠道/权限 | 决定“先企业主体、后角色/业务”的主流程 |
| Risk FSD | Hard Rules、AML/SIRON CDD/EDD、CCRIS/CTOS、AIP、UW、审批结果矩阵 | 进件提交前必须保证申请表、OCR、所有申请人/UBO/Owner/Guarantor 的关键字段完整 |
| CED FSD | CED 工作台、外部/内部数据整合、任务池、重查、Rework、风险暴露、信用备忘录 | Sales 进件必须保留字段来源和修改痕迹，支持 CED 退回后精准修正 |
| AIOCR FSD | 文件分类、OCR 抽取、交叉校验、必需材料清单、提交门禁 | 文件上传应按业务阶段切分，Application 和 Disbursement 材料不能混在一个重列表里 |
| Disbursement FRS | FIS、Dealer FBR/FBT、CRA 文件检查、Host Acceptance、CIF/Facility/Collateral 创建 | 放款前置依赖需要在进件时提示，但不应全部阻塞申请提交 |
| 企业提交信息.xlsx | 企业字段、字段来源、OCR/ETB 标识、Mandatory、Management & Shareholder | 字段定义应改成“来源分组 + 业务对象分组 + 必填条件” |
| 企业类型/角色/报送规则图片 | 12 类企业的法律地位、担保、签约、车辆注册、催收、Constitution Code | Entity Type 是整条流程的核心分叉点 |

## 3. 现有流程主要问题

| 问题 | 影响 | 建议 |
|---|---|---|
| 入口阶段先填渠道/分行/销售 | 销售还未确认客户主体，先处理银行内部归属，心智顺序不自然 | 入口只做“新建申请 + 企业识别”，渠道归属作为系统自动带出卡片 |
| Entity Type 由用户早期手选 | 用户可能不了解 SSM/ID Type 映射，容易选错，后续字段、UBO、报送都受影响 | 先输入 ID Type + ID No，系统根据 SSM/旧号规则推荐企业类型，用户确认 |
| UBO、Shareholder、Guarantor、Signatory 混在“关联人”概念里 | 开发和测试容易误判收入、报送、签约、担保责任 | 使用三维角色模型：债务身份、所有权身份、签约权力 |
| 风险/CED/放款依赖没有前置状态化 | 申请能提交，但后续被 CED/CRA 因字段、材料、签约、FIS 打回 | 增加“提交前准备度面板”，显示风险准备度、CED 准备度、放款准备度 |
| 文件上传按材料堆叠 | Sales 不清楚哪些是申请阶段必须、哪些是放款阶段补充 | 按业务阶段分 Tab：Application Submission、CED Condition、Disbursement |
| HP Line、FBR/FBT、HP+Plus 等特殊标签散落 | 特殊产品会影响风控、材料、定价、放款，但缺少统一显性提示 | 在案件顶部显示 Case Tags，并在相关步骤触发原因和后续影响 |

## 4. 推荐流程骨架

### Step 0：新建申请

用户点击 `New HP Application`，选择 `Application Type = Non-Individual`。系统生成 Application No.，从登录态带出 Officer、Branch、Region、默认 Channel Coverage。

页面只需要用户输入或确认：

| 字段 | 推荐处理 |
|---|---|
| Application Type | 默认为 Non-Individual |
| Officer / Branch | 从登录用户和权限主数据自动带出，只读展示 |
| Application No. | 系统生成 |
| Source | 可后置到车辆/Dealer 步骤；如果从 Dealer API/Lead 进入，则自动带出 |

### Step 1：锁定企业主体

这是进件真正的第一步。用户输入企业证件类型和号码，系统完成校验与识别：

| 动作 | 系统输出 |
|---|---|
| 输入 ID Type + ID No | 校验格式，推荐 Enterprise Type |
| 查询 CIF | 标记 ETB/NTB，带出历史信息 |
| 查询 HP Line | 命中白名单时弹窗选择普通案件或 HP Line |
| 调 Experian/SSM | 带出 Company Name、注册信息、董事/股东基础数据 |
| 建立主体锁 | ID Type + ID No 一旦确认后只读，错误则废弃申请重建 |

### Step 2：申请上下文确认

企业主体锁定后，再展示申请上下文：

| 信息 | 推荐交互 |
|---|---|
| Officer / Branch / Region | 系统带出，只读；如 Branch 已关闭，阻断提交并要求改派 |
| Channel Type | 默认由入口/Lead/Dealer 决定；Direct 可不要求销售手填 |
| Dealer Code | 只有 Indirect/Dealer 业务显示，放在车辆/Dealer 区域确认 |
| Sales Manager / Upline | 由组织架构和 Branch 默认规则自动路由 |
| Campaign / Source Code | 在产品/贷款方案步骤确认，因为它影响定价和 Greenlane |

### Step 3：企业画像补齐

按数据来源组织字段，而不是按历史表单堆叠：

| 分组 | 字段例子 | 来源 |
|---|---|---|
| 注册信息 | Company Name、New BR No、Old Reg No、Establishment Date、Constitution | Experian/SSM/ETB |
| 经营信息 | Nature of Business Group/Code、Customer Sector、Firm Type、Bumiputra/NRCC | 主数据/手填 |
| 税务与地域 | TIN、SST、Tourism Tax、Country/State of Operation、FEN Resident Status | 手填 + 校验 |
| 财务概况 | Annual Sales Turnover、Employees、Paid Up Capital、Primary Income Doc | 手填/OCR |
| 合规确认 | PEP/RCA、复杂结构、Nominee/Bearer Shares、Face-to-face onboarding | 系统筛查 + 手工确认 |

### Step 4：股权、UBO 与角色结构

系统先展示 Experian 抽取结果，再让销售补充无法自动穿透的信息。

关键设计：

- 用一个“企业结构工作台”同时呈现树形图和表格。
- 树形图用于理解穿透，表格用于开发落地和字段编辑。
- 每个节点都显示角色标签：Director、Shareholder、UBO Candidate、Guarantor、Signatory。
- 法人股东无法自动穿透时，节点上显示 `Drill-down Required`。
- UBO 只能是自然人，可系统推荐、用户选择或 Customize。

### Step 5：担保人与其他申请人

担保结构由 Entity Type 和产品规则驱动：

| 类型 | 推荐规则 |
|---|---|
| A Sdn Bhd | 通常要求 1-2 名董事 PG，系统提示但按业务配置决定是否强制 |
| B Berhad | 大型 Bhd 通常豁免 PG，展示豁免依据 |
| D Sole Prop | Owner 本人承担无限责任，不额外要求 PG |
| E Partnership | 全体 Partner/Owner 必须录入 |
| F/G/H PLT | 核心/全部 Partner 需加入 Guarantor，按政策配置 |
| J Society | 主要委员会成员 PG |
| K Government | 财政兜底，免担保，豁免 UBO |

每添加一个 Guarantor，系统触发 CIF、WT Whitelist、Income DB、App History、Pre-Consent 等查询。只有主申请人和 Guarantor 的收入进入 DSR/DSCR。

### Step 6：收入与风险指标

收入模块不建议放得过早。企业主体、角色和担保结构确认后，系统才知道谁的收入需要录入。

输出必须服务风险和 CED：

- Monthly Gross Income
- Monthly Net Income
- Fixed / Variable / Business / Rental / Others
- Total Commitment
- Current DSR / Internal DSR / Global DSR
- DSCR
- Minimum Disposable Income
- 收入来源标签：ETB Income、WT Income、Manual、OCR

### Step 7：车辆、Dealer 与产品方案

车辆和 Dealer 信息天然联动，应放在企业与担保结构后：

| 模块 | 关键点 |
|---|---|
| Vehicle | Make、Model、Variant、YOM、New/Used/Recond、MV/RB、BNM Green/Non-Green |
| Dealer/Seller | Direct 时可隐藏 Dealer；Dealer 时查询 Dealer status、FBR/FBT 资格 |
| Product | Conventional/Islamic、HP/IHP、HP Line、HP+Plus |
| Pricing | Rate、EIR、Campaign、Price Rule、Deviation |

FBR/FBT 只在放款前强相关，进件阶段只需要形成 Case Tag 和后续提醒。

### Step 8：材料上传与 OCR

文件上传分三类：

| Tab | 用途 |
|---|---|
| Application Submission | 身份、企业注册、收入、车辆、Consent 等进件材料 |
| CED Condition | CED 条件审批要求补件 |
| Disbursement | HP Agreement、Guarantor Agreement、VOC、Road Tax、Insurance、E-Hakmilik 等放款材料 |

提交到风险/CED 前必须完成：

- 文件格式/大小校验
- AI 分类确认
- OCR 抽取
- 关键字段人工确认
- 交叉校验
- 必需材料清单校验

### Step 9：提交前准备度

建议新增一个 Review 页面，按下游团队视角展示：

| 准备度 | 校验内容 |
|---|---|
| Risk Ready | Hard Rules 输入、AML/CDD/EDD 所需字段、CCRIS/CTOS 所需字段、收入/债务指标 |
| CED Ready | 企业主体、Guarantor、UBO、收入来源、OCR 差异、材料完整性、风险标签 |
| Disbursement Ready | Signatory、Manual Acceptance、Host Acceptance 关键字段、FIS/Dealer/FBR/FBT 提醒、车辆注册/签约主体 |
| BNM Ready | Constitution Code、Entity Category、ID Type、TIN、UBO、CCRIS 报送字段 |

## 5. 与风险、CED、放款的关键联动

| 下游环节 | 依赖进件字段/状态 | 产品要求 |
|---|---|---|
| Hard Rules | 年龄、国家/地区、受制裁国家、申请材料完整性 | 在提交风险前跑预校验 |
| AML CDD/EDD | 主申请人、Guarantor、Owner、UBO、Signatory 等所有相关方信息 | 每个相关方要有 AML 状态和结果可追溯 |
| CCRIS/CTOS | 主申请人、Guarantor 的 ID、地址、收入、贷款信息 | CCRIS 失败时按配置进入 CED，不让销售无感继续 |
| UW/AIP | 收入、负债、车辆、贷款方案、风险标签 | Review 页面显示缺失项 |
| CED | OCR 结果、收入计算、风险暴露、Deviation、Exception、材料 | 保留字段来源和销售修改记录 |
| CED Rework | 地址、姓名、Guarantor、收入、材料等可能修改 | Rework 后必须重跑相关规则，并保留前后差异 |
| Acceptance | 签约人、担保人、协议、手动/电子接受规则 | Non-Individual 默认手动签署，不误导电子签 |
| Disbursement | FIS、Dealer FBR/FBT、VOC、Road Tax、Insurance、Host Acceptance | 提前提示，放款阶段再按 CRA 规则强校验 |
