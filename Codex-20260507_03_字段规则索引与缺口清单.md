## 2. 当前明确的流程修正

| 原 V2 设计 | Codex 修正 |
|---|---|
| 阶段零先填渠道、Branch、Dealer、客户类型、Entity Type | Application Details 只建申请壳；Company Basic Information 先锁定企业主体；Branch/Officer 自动带出 |
| Entity Type 早期人工选择 | 由 ID Type + ID No + SSM/Experian/HOST 识别推荐，用户确认 |
| 独立 12 阶段流程 | 并入现网 Navigation，作为 Applicant Type = Non-Individual 的分支 |
| 后续风险/放款展开较多 | 只保留进件准备度和概念链路 |
| 深色独立导航原型 | 改为继承现网白底 Navigation 卡片 |

## 3. 仍需确认的缺口

| 优先级 | 问题 | 影响 |
|---|---|---|
| 高 | A 类 Sdn Bhd 是否强制 1-2 名董事 PG | 影响 Guarantor Information 必填规则 |
| 高 | F/G/H PLT 是全部 Partner 还是核心 Partner 作为 Guarantor | 影响担保人强制校验 |
| 高 | Bhd 上市/大型客户 UBO 豁免或 Top 5 简化标准 | 影响 UBO Identification |
| 高 | C Branch / Foreign LLP / East MY SE 的 Constitution 归属 | 影响 BNM/CCRIS 报送 |
| 中 | Emergency Contact 在企业场景是否改为 Contact Person | 影响导航继承 |
| 中 | Dealer 信息应该在 Application Details 还是 Collateral & Seller 中确认 | 影响交互位置 |
| 中 | HP Line 命中后是否允许普通案件继续，是否强制记录原因 | 影响弹窗和审计 |
| 中 | Non-Individual 是否完全不使用 E-Consent/E-Acceptance | 影响导航中 Identity/Consent 状态 |
| 低 | 各收入计算器字段、公式和 OCR 字段映射 | 影响开发细化 |
| 低 | 必需材料清单配置粒度 | 影响 OCR/Document 模块 |

## 4. 开发使用方式

开发不要只看 Codex 精炼文档。正确方式是：

1. 用 Codex 文档理解页面顺序和并入现网的方式。
2. 用 V2 和 Excel 查每个字段、枚举、必填、OCR、ETB、校验。
3. 用原始 FSD 处理细节争议。
4. 用缺口清单找业务/合规/技术确认。
